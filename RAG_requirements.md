# 企业级RAG知识库项目 前期需求技术\+后期落地要求（优化版）

本文档结合项目「开发前期」核心场景，拆分「前期需求与技术适配」和「后期落地与优化要求」，兼顾本地轻量验证与后期企业级部署，适配求职项目落地节奏，整合前期技术选型结论，重点突出前期开发重点、规避前期踩坑，明确后期落地标准，可直接作为全流程开发参考。

# 第一部分：前期核心（开发前期·需求\+技术适配）

核心目标：**快速完成本地轻量验证，跑通核心流程，聚焦功能落地，不纠结生产级配置**，优先实现文档上传、检索、多轮问答核心链路，为后期切换企业级组件奠定基础。

## 一、前期核心需求（聚焦可验证，简化非必要功能）

### 1\.1 必做功能（前期优先落地，确保本地可演示）

- 用户与权限管理：简化版RBAC权限（区分管理员/普通用户），暂不实现多租户（后期扩展），控制文档查看范围；

- 文档管理：文档上传（支持PDF/Word/TXT常见格式）、解析、清洗、分块、本地持久化存储（无需MinIO）；

- 检索功能：向量检索\+BM25多路召回，集成Rerank重排序，确保检索精度（核心验证点）；

- 问答功能：多轮会话、指代消解（优先用会话历史\+Prompt实现），答案带文档溯源，无幻觉；

- 会话管理：会话持久化（刷新不丢失）、新建/切换/删除会话，暂用内存缓存（后期替换Redis）；

- 基础前端：极简可用版（登录、会话列表、聊天窗口、文档上传），无需复杂UI，能演示核心流程即可；

- 本地调试：支持本地启动、快速调试，无需Docker部署（后期补充）。

### 1\.2 可选功能（前期可暂不做，后期补充加分）

- Query改写：暂不实现，待核心流程跑通后，作为优化项补充（提升检索精度，丰富简历亮点）；

- 文档批量上传与异步处理：前期实现单文件上传、同步解析即可，异步处理后期补充；

- 效果评估、检索日志：前期无需开发，后期优化阶段补充。

### 1\.3 禁止做的功能（前期规避过度设计，聚焦核心）

- 不做复杂语义分块、LLM智能分块、GraphRAG（场景不匹配，过度设计，前期用基础分块即可）；

- 不做Agent核心能力（自主任务规划、工具调用、流程状态机，与第一个项目重复，且非本项目核心）；

- 不做复杂前端（移动端适配、华丽动画、数据看板，极简可用即可）；

- 不做长时间记忆（用户偏好归纳、人格沉淀，企业RAG无需此功能，前期聚焦会话上下文衔接即可）；

- 不引入复杂中间件（Milvus、Redis、MinIO，前期用轻量替代方案）。

## 二、前期技术选型（适配本地轻量开发，快速验证）

### 2\.1 核心框架选型（优先方案1，快速落地；方案2后期优化）

- 优先方案：纯LangChain（稳妥、快速落地，前期核心选择）

    - RAG核心：LangChain（分块、检索、会话管理、Prompt编排）；

    - 分块：RecursiveCharacterTextSplitter（固定参数：chunk\_size=700，chunk\_overlap=150，分隔符\[\&\#34;\\n\\n\&\#34;, \&\#34;\\n\&\#34;, \&\#34;。\&\#34;, \&\#34;？\&\#34;, \&\#34;！\&\#34;\]）；

    - 检索：手动组装BM25\+向量检索，集成Rerank模型（前期优先实现，验证检索精度）；

    - 优势：技术栈统一，全链路可控，开发速度快，适合前期快速验证；

    - 不足：检索相关功能需手动组装，但前期代码量可控，后期可复用。

- 后期优化方案：LangChain \+ LlamaIndex 结合（前期不引入，避免增加调试成本）
        

    - 分工：后期将检索层替换为LlamaIndex（文档解析、智能分块、混合检索、Rerank），LangChain保留上层业务逻辑；

    - 优势：提升检索精度，增加简历亮点，适配企业高级RAG架构；

    - 实施时机：前期核心流程跑通、本地验证完成后，作为优化项引入。

### 2\.2 前期轻量技术选型（适配本地开发，无需复杂部署）

|模块|前期技术选型（本地轻量）|核心作用|关键参数/配置|
|---|---|---|---|

|检索模型|Embedding：BGE\-base\-zh；Rerank：BGE\-reranker\-base|生成向量、优化检索结果排序，确保前期检索精度|中文适配性强，轻量、速度快，无需本地部署复杂模型|
|LLM大模型|Qwen / Llama 3 / ChatGLM（优先调用API，本地部署可选）|指代消解、生成答案，后期补充Query改写|无需微调，直接调用API，降低前期调试成本|
|向量库|Chroma（Python库，本地文件存储）|存储文档向量，支持快速相似度检索，无需部署|适配LangChain，本地文件持久化，无需启动容器|

## 三、前期开发核心要领（落地关键，规避前期踩坑）

### 3\.1 分块策略（前期核心，确保检索基础精度）

- 前期采用纯LangChain的RecursiveCharacterTextSplitter，优先按段落、句子软切，避免硬断句子；

- 固定参数：chunk\_size=700，chunk\_overlap=150，分隔符按中文习惯配置，不做复杂调整；

- 关键：不做字符硬切，保留句子完整性，避免语义断裂，前期无需追求复杂分块策略。

### 3\.2 检索流程（前期必做，确保核心功能可验证）

1. 用户提问 → 暂不做Query改写（后期补充）；

2. Embedding模型将问题转为向量；

3. 多路召回：向量检索（召回10条）\+ BM25关键词检索（召回10条）；

4. Rerank模型对结果打分、去重、排序，保留最相关的前3\-5条（必须做，否则检索精度不足）；

5. 将检索结果传入LLM，结合最近5轮会话历史，生成带溯源的答案。

### 3\.3 多轮会话与指代消解（前期简化，确保可用）

- 会话持久化：会话历史存入PostgreSQL，用内存缓存最近5轮，避免Token过长；

- 指代消解：无需复杂算法，将最近5轮会话历史传入Prompt，让LLM自动理解代词（这个、那、它）；

- 关键：前期无需优化过多，确保上下文衔接自然，无明显断层即可。

### 3\.4 前期避坑重点（重中之重，避免走弯路）

- 框架选型：前期不引入LlamaIndex、LangGraph等复杂框架，避免增加调试成本，先跑通纯LangChain流程；

- 中间件：不使用Milvus、Redis、MinIO，用Chroma、SQLite、本地文件替代，简化环境配置；

- 功能开发：不追求“一步到位”，优先实现核心链路，可选功能后期补充，避免过度设计；

- 调试：本地开发优先启用热重载，及时排查接口、检索、问答中的问题，不积累bug；

- 代码：前期保证简洁、可复用，重点注释核心逻辑，为后期切换企业级组件做好铺垫。

### 3\.5 前期实施步骤（按顺序落地，不混乱）

1. 环境搭建：安装Python依赖（LangChain、FastAPI、Chroma等）、Node\.js依赖（前端），无需配置Docker；

2. 基础开发：实现简化版用户登录、权限校验，文档上传与本地文件存储；

3. 核心开发：文档解析、分块、向量化（Chroma），构建检索链路（混合检索\+Rerank）；

4. 会话开发：会话持久化（PostgreSQL）、多轮上下文管理、指代消解，实现基础问答功能；

5. 前端开发：极简界面（登录、聊天、文档上传），对接后端接口，确保可演示；

6. 本地测试：准备测试问句集，验证检索精度、会话衔接，解决幻觉、检索不准等基础问题；

7. 代码整理：优化核心逻辑，注释清晰，为后期切换企业级组件、补充功能做好准备。

# 第二部分：后期要求（功能优化\+企业级落地）

核心目标：**从本地轻量验证，升级为企业级可部署、可复用系统**，补充前期简化的功能，切换企业级组件，满足工程化部署要求，丰富简历亮点，适配企业生产场景。

## 一、后期功能优化要求（补充前期简化项，提升系统竞争力）

- 权限与多租户：补充多租户功能，按租户ID隔离文档、会话、用户，完善RBAC细粒度权限（管理员/普通用户/只读用户）；

- Query改写：实现轻量化Query改写（仅用最近1\-2轮历史），将口语化、带代词的问句转为标准检索问句，提升检索精度；

- 文档处理：补充文档批量上传、异步解析功能，避免前端阻塞，提升用户体验；

- 效果优化：增加LLM自动打分功能（检索相关性、答案准确性、完备性），便于优化检索效果；

- 日志与监控：添加检索日志、用户操作日志，记录检索行为、检索结果，便于后期优化与问题排查；

- 异常处理：完善异常捕获（文档解析失败、检索失败、LLM调用失败等），返回友好提示，提升系统稳定性。

## 二、后期技术升级要求（切换企业级组件，适配生产场景）

### 2\.1 核心框架升级（可选，提升简历亮点）

- 切换为LangChain \+ LlamaIndex融合架构：LlamaIndex负责底层检索（文档解析、智能分块、混合检索、Rerank），LangChain负责上层业务（权限、会话、Prompt、前端接口、服务化）；

- 核心要求：确保架构兼容，前期纯LangChain代码可复用，仅替换检索层，不重写核心业务逻辑。

### 2\.2 组件升级（从本地轻量→企业级）

|模块|前期轻量组件|后期企业级组件|升级要求|
|---|---|---|---|
|向量库|Chroma|Milvus|适配LangChain/LlamaIndex，实现向量数据迁移，确保检索精度不变|
|业务数据库|SQLite|PostgreSQL|实现数据迁移，支持多租户，确保会话、用户、文档元数据不丢失|
|缓存|内存缓存|Redis|缓存最近5轮会话、用户权限，设置合理过期时间，提升访问速度|
|文件存储|本地文件系统|MinIO|实现源文件迁移，支持文件上传、下载、删除，适配企业存储需求|

## 三、后期工程化部署要求（企业级标准，可演示、可迁移）

- Docker Compose部署：编写完整docker\-compose\.yml，编排所有组件（FastAPI、PostgreSQL、Redis、Milvus、MinIO、前端），实现一键启动；

- 环境配置：编写完整\.env文件，统一管理所有组件的地址、密钥、模型路径，便于环境切换；

- 服务化：优化FastAPI后端，支持服务化启动，适配企业部署规范，添加健康检查接口；

- 可迁移性：确保Docker镜像可复用，部署步骤清晰，可在不同环境快速启动，便于面试演示；

- 稳定性：测试全流程部署，排查容器网络、端口、挂载等问题，确保系统稳定运行，无明显bug。

## 四、后期优化与面试准备要求

- 检索优化：调整分块参数、检索权重、Rerank模型参数，提升检索精度，降低幻觉率；

- 性能优化：优化文档解析、检索速度，减少LLM调用耗时，提升用户体验；

- 简历适配：整理架构升级、功能优化的核心亮点，完善面试话术，突出工程化思维与问题解决能力；

- 演示准备：确保部署后的系统可完整演示（文档上传→检索→多轮问答→会话管理），准备测试用例，应对面试演示。

## 五、后期实施步骤（按顺序升级，不混乱）

1. 功能补充：实现Query改写、多租户、批量上传、日志等前期未做的功能；

2. 组件升级：逐步替换Chroma→Milvus、SQLite→PostgreSQL、内存缓存→Redis、本地文件→MinIO，确保数据迁移无误；

3. 架构优化（可选）：引入LlamaIndex，替换检索层，优化检索精度；

4. 部署配置：编写docker\-compose\.yml、\.env文件，配置所有组件，测试容器启动与组件连接；

5. 全流程测试：验证部署后的系统功能、性能、稳定性，解决部署过程中的问题；

6. 面试准备：完善项目文档、演示用例、面试话术，确保能清晰讲解项目全流程、技术选型理由、优化点。
